2 ЗАВДАННЯ
     Метою роботи є дослідження архітектури складної програмної системи та аналіз застосованих архітектурних підходів.
     У межах роботи необхідно:
* виконати аналіз архітектури програмної системи;
* визначити архітектурний стиль;
* описати основні компоненти;
* проаналізувати взаємодію підсистем;
* оцінити масштабованість та переваги архітектурних рішень;
* підготувати презентацію та відеодоповідь;


3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Загальні відомості про систему
     Назва системи: Spotify
     Розробник: Spotify Technology S.A.
     Призначення системи:
     Spotify є однією з найбільших стрімінгових платформ для прослуховування музики, подкастів та аудіокниг через мережу Інтернет. Система орієнтована як на звичайних користувачів, так і на виконавців, власників контенту та рекламодавців.
     Основними функціями системи є:
* пошук музичних композицій, альбомів та виконавців;
* потокове відтворення аудіоконтенту;
* створення та керування плейлистами;
* формування персональних рекомендацій;
* завантаження музики для офлайн-прослуховування;
* збір статистики прослуховувань;
* розміщення рекламного контенту.
     Платформа підтримує роботу через веббраузери, мобільні додатки та настільні клієнти.
     3.2 Архітектурний стиль системи
     Spotify використовує мікросервісну архітектуру з елементами подієво-орієнтованого підходу.
     У перших версіях система використовувала більш монолітну архітектуру, однак зі збільшенням кількості користувачів та сервісів було виконано перехід до мікросервісної моделі.
     Основними причинами вибору мікросервісної архітектури стали:
* можливість незалежної розробки окремих компонентів;
* спрощення масштабування окремих сервісів;
* підвищення відмовостійкості системи;
* можливість використання різних технологій для різних сервісів;
* спрощення розгортання та оновлення програмного забезпечення.
     Кожен мікросервіс виконує окрему функцію та взаємодіє з іншими сервісами через API або системи обміну повідомленнями.
     3.3 Основні компоненти системи
     Інтерфейс користувача:
     Інтерфейс представлений вебверсією, мобільними застосунками для Android та iOS, а також десктопними клієнтами.
     Користувач взаємодіє із системою через графічний інтерфейс, який забезпечує пошук контенту, керування плейлистами та відтворення музики.
     Серверна частина:
     Серверна частина складається з великої кількості мікросервісів, кожен з яких відповідає за окрему функцію:
* керування користувачами;
* авторизацію;
* музичний каталог;
* рекомендації;
* обробку платежів;
* аналітику.
     Рівень бізнес-логіки
     Рівень бізнес-логіки реалізує основні правила роботи системи:
* перевірку прав доступу;
* формування рекомендацій;
* керування підписками;
* аналіз активності користувачів;
* генерацію персоналізованого контенту.
     Рівень зберігання даних:
     Для зберігання інформації використовуються розподілені бази даних та сховища.
     Зберігаються:
* профілі користувачів;
* музичний каталог;
* плейлисти;
* історія прослуховувань;
* статистичні дані.
     Допоміжні сервіси
     До допоміжних сервісів належать:
* системи моніторингу;
* балансувальники навантаження;
* кеш-сервери;
* сервіси логування;
* системи аналітики.
     3.4 Взаємодія компонентів
     Коли користувач запускає музичну композицію, клієнтський застосунок надсилає запит до API-шлюзу. API-шлюз перенаправляє запит до відповідного сервісу авторизації для перевірки прав доступу та до сервісу музичного каталогу для отримання інформації про трек.
     Після успішної перевірки сервіс потокового відтворення надсилає аудіодані клієнту. Паралельно інформація про прослуховування передається до сервісів аналітики та рекомендацій.
     Для взаємодії між компонентами використовуються протоколи HTTP/HTTPS, REST API та внутрішні механізми обміну повідомленнями.
     UML-діаграма компонентів:

     Рисунок 3.1 – Діаграма компонентів стрімінгової платформи Spotify
     Клієнтський застосунок є точкою входу користувача до системи. Усі запити надходять через API Gateway, який маршрутизує їх до відповідних мікросервісів. Кожен сервіс взаємодіє зі своїми даними та виконує спеціалізовані функції.
     3.5 Масштабованість та надійність
     Для забезпечення роботи з сотнями мільйонів користувачів Spotify використовує сучасні механізми масштабування та забезпечення надійності.
     Балансування навантаження:
     Запити користувачів розподіляються між декількома серверами за допомогою балансувальників навантаження.
     Відмовостійкість:
     Мікросервіси працюють незалежно один від одного. Відмова одного сервісу не призводить до повної зупинки системи.
     Кешування:
     Для прискорення доступу до популярних даних використовуються розподілені кеші.
     Реплікація:
     Бази даних реплікуються між кількома серверами та дата-центрами для забезпечення доступності даних.
     Контейнеризація та хмарна інфраструктура:
     Spotify активно використовує контейнеризацію та автоматизовані засоби оркестрації сервісів. Це дозволяє швидко масштабувати систему відповідно до поточного навантаження.
     3.6 Приклади програмної реалізації
     Spotify надає Web API, який дозволяє стороннім розробникам отримувати інформацію про виконавців, альбоми, треки та плейлисти.
     Приклад HTTP-запиту для отримання інформації про трек:
     GET https://api.spotify.com/v1/tracks/{id}
     Authorization: Bearer ACCESS_TOKEN
     Приклад відповіді API:
{
  "id": "4uLU6hMCjMI75M1A2tKUQC",
  "name": "Never Gonna Give You Up",
  "duration_ms": 213000,
  "popularity": 78
}
     Приклад використання API мовою Python:
import requests

url = "https://api.spotify.com/v1/tracks/4uLU6hMCjMI75M1A2tKUQC"

headers = {
    "Authorization": "Bearer ACCESS_TOKEN"
}

response = requests.get(url, headers=headers)

print(response.json())

     Наведений приклад демонструє принцип сервісно-орієнтованої взаємодії, коли клієнт отримує необхідні дані через API без прямого доступу до внутрішніх компонентів системи.


4 ВИСНОВКИ
     У ході виконання роботи було досліджено архітектуру стрімінгової платформи Spotify та проаналізовано основні архітектурні підходи, що забезпечують її ефективне функціонування. Розглянуто призначення системи, її основні функції та особливості роботи з великою кількістю користувачів і мультимедійного контенту.
     Під час аналізу було встановлено, що Spotify використовує мікросервісну архітектуру, яка дозволяє розподілити функціональність системи між незалежними сервісами. Такий підхід забезпечує високу масштабованість, спрощує супровід програмного забезпечення та дозволяє незалежно розгортати й оновлювати окремі компоненти системи.
     Було досліджено структуру основних компонентів платформи, зокрема клієнтські застосунки, API Gateway, сервіси авторизації, потокового відтворення, рекомендацій, аналітики та зберігання даних. Також розглянуто механізми їх взаємодії та обміну даними за допомогою API та мережевих протоколів.
     Отже, проведене дослідження показало, що архітектурні рішення, використані в Spotify, є ефективним прикладом побудови сучасної високонавантаженої програмної системи. Використання мікросервісної архітектури забезпечує гнучкість, масштабованість, відмовостійкість та можливість подальшого розвитку платформи.
