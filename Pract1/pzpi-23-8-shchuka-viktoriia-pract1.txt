3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Загальні відомості про шаблони (патерни) проєктування
     Шаблони (патерни) проєктування — це типові перевірені рішення поширених проблем, що виникають під час розробки програмного забезпечення. Вони не є готовим кодом, а являють собою рекомендації щодо організації взаємодії між класами та об'єктами. Використання патернів дозволяє підвищити гнучкість, масштабованість та підтримуваність програмних систем.
     Найбільш відомою класифікацією є класифікація GoF (Gang of Four), яка поділяє патерни на три групи:
* породжувальні (Creational) — відповідають за створення об'єктів;
* структурні (Structural) — визначають способи побудови складних структур із класів та об'єктів;
* поведінкові (Behavioral) — описують взаємодію між об'єктами та розподіл відповідальності.
     Патерни проєктування використовуються для стандартизації процесу розробки, спрощення повторного використання коду та підвищення якості програмних продуктів.
     3.2 Опис обраного патерна
     Назва патерна: Prototype (Прототип)
     Тип шаблона (патерна): породжувальний (Creational Pattern)
     Основне призначення:
     Патерн Prototype дозволяє створювати нові об'єкти шляхом копіювання вже існуючих екземплярів без залежності від їх конкретних класів. Замість створення нового об'єкта через конструктор використовується механізм клонування.
     Головна ідея полягає в тому, що об'єкт сам відповідає за створення власної копії. Це особливо корисно, коли процес створення об'єкта є складним або ресурсомістким.
     Передумови застосування:
Використання патерна Prototype є доцільним у таких випадках:
* створення об'єкта потребує значних обчислювальних ресурсів;
* необхідно приховати складну логіку створення об'єктів;
* система повинна працювати з великою кількістю схожих об'єктів;
* потрібно створювати копії об'єктів під час виконання програми;
* класи об'єктів невідомі заздалегідь або можуть динамічно змінюватися.
     Переваги:
* зменшує залежність від конкретних класів;
* прискорює створення складних об'єктів;
* дозволяє створювати об'єкти під час виконання програми;
* спрощує додавання нових типів об'єктів;
* зменшує дублювання коду створення екземплярів.
     Недоліки:
* складність реалізації глибокого копіювання об'єктів;
* можливі проблеми з копіюванням вкладених посилань;
* необхідність реалізації механізму клонування для кожного класу;
* збільшення складності системи при роботі зі складними структурами об'єктів.
     3.3 Структура шаблона (патерна)
     Опис структурних частин:
     Prototype – інтерфейс або абстрактний клас, який оголошує метод clone() для створення копії об'єкта.
     ConcretePrototype – конкретний клас, що реалізує інтерфейс Prototype та визначає спосіб копіювання власних даних.
     Client – клієнтський код, який створює нові об'єкти шляхом виклику методу clone(), не знаючи деталей реалізації конкретних класів.
     Опис взаємодії об'єктів:
1. Клієнт має посилання на існуючий об'єкт-прототип. 
2. Клієнт викликає метод clone(). 
3. Метод clone() створює новий екземпляр відповідного класу. 
4. Під час створення копії значення полів вихідного об'єкта переносяться до нового об'єкта. 
5. Новий екземпляр повертається клієнту для подальшого використання.
     UML-діаграма:
     На UML-діаграмі представлено основні учасники патерна Prototype та їх взаємодію (див. рис. 3.1).
* Prototype — інтерфейс, який оголошує метод clone(). Цей метод відповідає за створення копії об'єкта. 
* ConcretePrototype — конкретний клас, що реалізує інтерфейс Prototype. Він містить власні поля даних (field1) та реалізує механізм копіювання об'єкта. 
* SubclassPrototype — похідний клас від ConcretePrototype, який додає власні поля (field2) та перевизначає логіку клонування для коректного копіювання як успадкованих, так і власних даних. 
* Client — клієнтський код, який працює через інтерфейс Prototype і створює нові об'єкти шляхом виклику методу clone().

     3.4 Реалізація шаблона (патерна)
     Для демонстрації роботи патерна Prototype використано мову програмування Java. Реалізація полягає у створенні інтерфейсу для клонування об'єктів та конкретного класу, який реалізує механізм створення копій власних екземплярів.
     Створення базового інтерфейсу
     На першому етапі створюється інтерфейс Prototype. Він містить метод clone(), який повинен повертати копію об'єкта. Завдяки цьому клієнтський код може працювати з будь-якими класами, що реалізують даний інтерфейс.
      public interface Prototype {
          Prototype clone();
     }
     Інтерфейс визначає єдиний контракт для всіх класів-прототипів і не залежить від конкретної реалізації об'єктів.
     Реалізація конкретного класу:
     Далі створюється клас Car, який реалізує інтерфейс Prototype. Клас містить характеристики автомобіля та реалізує метод clone(), що створює новий об'єкт із такими самими значеннями полів.
      public class Car implements Prototype {
      
          private String model;
          private String color;
      
          public Car(String model, String color) {
              this.model = model;
              this.color = color;
          }
      
          @Override
          public Prototype clone() {
              return new Car(model, color);
          }
      
          @Override
          public String toString() {
              return "Car{" +
                      "model='" + model + '\'' +
                      ", color='" + color + '\'' +
                      '}';
          }
      }
     
     У методі clone() створюється новий об'єкт класу Car, якому передаються значення полів поточного екземпляра. Таким чином формується повна копія об'єкта.
     Демонстрація використання
     Для перевірки роботи патерна створюється початковий об'єкт, після чого виконується його клонування за допомогою методу clone().
      public class Main {
      
          public static void main(String[] args) {
      
              Car original = new Car("Tesla Model 3", "White");
      
              Car copy = (Car) original.clone();
      
              System.out.println("Оригінал: " + original);
              System.out.println("Копія: " + copy);
          }
      }
     
     У результаті виконання програми буде створено два незалежні об'єкти: оригінал та його копію. Обидва об'єкти матимуть однакові значення властивостей, але займатимуть різні області пам'яті.
     Результат виконання програми:
     Оригінал: Car{model='Tesla Model 3', color='White'}
     Копія: Car{model='Tesla Model 3', color='White'}
     У наведеному прикладі створення нового об'єкта відбувається не через безпосередній виклик конструктора клієнтом, а шляхом клонування вже існуючого екземпляра. Це дозволяє приховати складність створення об'єктів та забезпечує незалежність клієнтського коду від конкретних класів.
     Патерн Prototype особливо корисний у випадках, коли створення об'єкта є дороговартісною операцією або коли необхідно швидко створювати велику кількість схожих екземплярів із мінімальними змінами параметрів.
     Основні етапи реалізації
1. Створити інтерфейс Prototype із методом clone().
2. Реалізувати конкретні класи, які підтримують клонування.
3. Визначити логіку копіювання полів об'єкта.
4. Використовувати метод clone() для створення нових екземплярів.
5. Застосовувати отримані копії незалежно від оригінальних об'єктів.
     Таким чином, патерн Prototype дозволяє ефективно створювати нові об'єкти шляхом копіювання існуючих екземплярів, що особливо корисно для складних або ресурсомістких об'єктів.
4 ВИСНОВКИ
     У ході виконання роботи було досліджено шаблони проєктування як один із важливих інструментів об'єктно-орієнтованого програмування. Розглянуто їх призначення, класифікацію за GoF та особливості застосування під час розробки програмного забезпечення.
     Особливу увагу було приділено породжувальному патерну Prototype, який призначений для створення нових об'єктів шляхом копіювання вже існуючих екземплярів. Було проаналізовано його структуру, основні компоненти та принцип взаємодії між ними. Визначено переваги та недоліки патерна, а також умови, за яких його використання є найбільш ефективним.
     У роботі було побудовано UML-діаграму патерна Prototype та виконано його програмну реалізацію мовою Java. Розроблений приклад продемонстрував механізм клонування об'єктів за допомогою методу clone(), що дозволяє створювати нові екземпляри без безпосереднього використання конструкторів.
     Отримані результати підтверджують, що патерн Prototype сприяє зменшенню залежності клієнтського коду від конкретних класів, спрощує процес створення складних об'єктів та підвищує гнучкість програмної системи. Використання даного патерна є доцільним у випадках, коли створення об'єктів потребує значних ресурсів або коли необхідно працювати з великою кількістю схожих екземплярів.
